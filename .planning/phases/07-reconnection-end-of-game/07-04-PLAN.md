<threat_model>
## Trust Boundaries

| Boundary | Description |
|----------|-------------|
| Browser → Socket.IO submit_vote | Client sends continue_game boolean; server validates player_id belongs to session and deduplicates votes |
| PostGame receives final_scores from server | Data rendered as-is; no user-supplied input; XSS risk limited to player_name from server side |

## STRIDE Threat Register

| Threat ID | Category | Component | Disposition | Mitigation Plan |
|-----------|----------|-----------|-------------|-----------------|
| T-07-04-01 | Tampering | submit_vote double submission | accept | Server deduplicates (plan 07-02); client also guards with myVoteSubmitted state — defense in depth |
| T-07-04-02 | Information Disclosure | final_scores payload contains all player IDs and scores | accept | No PII; scores are game data intended to be visible to all players at game end |
| T-07-04-03 | Denial of Service | chat auto-scroll blocking user reading | mitigate | userScrolled ref tracks manual scroll; auto-scroll only fires when user is at bottom (ChatPanel.tsx implementation detail) |
| T-07-04-04 | Spoofing | player_name rendered in chat messages | mitigate | player_name is set at join time, validated by server; chat payload comes from server via trusted Pyro5 callback path |
</threat_model>
